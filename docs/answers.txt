(I) Inheritance is where a child class absorbs fields and methods from it's parent class, that can also be used on the child. This reduces redundancy, so methods do not have to be re-written for each child class
(II) Polymorphism is where objects of different types, can all be treated as instances of the parent class. For instance, the parent class Shape has a method draw(). Each of the child classes Triangle, Square, and Circle are different objects, but can define their own definition of draw() within their own class, and still be referenced as an instance object of the parent Shape within the main program.
(III) Encapsulation is the idea of bundling all variables/fields and methods together in a single unit referred to as a class.

9.
In the "Dictator and Lieutenants" workflow, all developers work from one centralized "blessed repository", and push to lieutenant repositories. Each lieutenant repository accepts pushes from a subset of developers, and solves any arising conflicts. Each lieutenant repository then pushes to the dictator repository, which solves conflicts between the lieutenant pushes. And finally the dictator pushes to the "blessed repository" for the developers to pull from again.
The "Integration manager" workflow still uses the centralized "blessed repository" that the developers pull from. But each developer pushes their code to their own "public repository". Then a centralized integration manager pulls from each developer's public repository, resolves any conflicts, and pushes the final product to the central "blessed repository", where the developers can pull from again.

10.
A team of 100 developers would benefit from the "Dictator and Lieutenants" workflow more, because each of the lieutenant repositories would be able to divide up the conflict resolution work, and the developers would not have to worry about that themselves. Where in the Centralized workflow, coordinating tasks between 100 developers, and resolving any conflicts between each other's commits would feel nearly impossible.
